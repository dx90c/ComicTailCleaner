### **ComicTailCleaner 開發計畫與交接文件 (v17.1.0 發布版)**

**致所有協作者 (包括 AI)：**

您好，歡迎加入 ComicTailCleaner 專案。本文件記錄了專案的架構演進、當前穩定版本的狀態，以及未來的開發與維護方向。

**目前狀態**：**v17.1.0 (Stable)** 已發布。此版本完成了核心比對效能優化、掃描與快取可觀察化、EH/EMM 數據庫工具穩定化，以及發布版打包流程的安全收斂。

---

### **一、開發路線圖 (Development Roadmap)**

本專案採用語義化版本號，但內部引擎版本號不一定與 App 發布版本同步。對外發布時以 `APP_VERSION` 為準。

#### **v15.x 系列 (已完成)**
*   **v15.0.0**: 確立了 src 目錄結構，修復了檔案掃描與進度條的核心缺陷，確保了打包後的路徑穩定性。

#### **v16.x 系列 (架構重構與快取遷移 - 已完成)**

*   **v16.0.2 (已完成)**:
    *   **核心重構**: **快取系統 SQLite 化**。棄用 JSON，改用 SQLite (WAL 模式) 管理圖片與資料夾快取，降低大數據量下的 I/O 瓶頸與併發寫入風險。
    *   **外掛架構**: **共享核心引擎**。`manga_deduplication` 外掛直接呼叫核心引擎公開 API，避免重複運算。
    *   **設定管理**: 實作 `get_default_config` 機制，主程式可自動偵測並合併外掛預設設定。
    *   **GUI 模組化**: 將龐大的 `gui.py` 拆分為 `gui/` 套件，獨立出 `settings_window.py` 與 `tooltip.py`。
    *   **EH 工具修復**: 完善 `eh_database_tools` 的備份輪替、CSV 輸出與自動化流程防禦。

#### **v17.x 系列 (效能、安全網與發布流程 - 當前階段)**

*   **v17.0.0 ~ v17.1.0 (已完成 - 當前版本)**:
    *   **相似度比對加速**: Phase A/B/C/E 大量比對流程導入 NumPy 向量化，改善大型資料夾的廣告比對、互相比與 QR 似廣流程效能。
    *   **廣告比對補強**: 針對 LSH 候選漏抓情境加入 Grid-Block Fallback，並補上 0-hash、自身比對與候選數量控制等安全邊界。
    *   **掃描與快取可觀察化**: 任務完成後顯示快取命中、重算、淘汰與重掃資料夾數量；設定頁快取清理入口加入更明確的警告與 SQLite family 清理策略。
    *   **Everything SDK 整合**: 主掃描流程可使用 Everything IPC 加速檔案發現；Everything 未執行或 SDK 不可用時會自動回退標準掃描。
    *   **EH/EMM 數據庫工具收斂**: 修正 EMM 顯示狀態同步、metadata deleted 重複計數、空資料夾處理邊界、Pagediff 修復與備份輪轉。
    *   **介面與日誌改善**: 狀態列改為穩定版面，EH 摘要接入完成狀態；任務結束 LOG 輸出固定 `# RECAP` 摘要，info/error log 加入大小輪轉。
    *   **安全網建置**: 建立一鍵指紋巡檢 Level 1/2/3，用於更新前後快速比對功能指紋、EH DB 狀態與 LOG 摘要。
    *   **發布流程收斂**: 建立乾淨 source 打包規則與 PyInstaller 打包腳本，排除私人 config、EH/EMM DB、cache、log、scratch、UpdateRecords 與測試殘留。
    *   **打包版穩定化**: 修復單檔 exe 中 UnRAR、Everything DLL、pyzbar、websocket 與 EMM subprocess 的 runtime 問題。

#### **未來展望 (v17.x 後續 / v18.x)**

*   **EH-FEAT 後續**:
    *   只有在 `[EH timing]` LOG 證明瓶頸後，才評估 EH 同步第二階段優化，例如更深的 Everything SDK 應用或 EhTag cache。
    *   任何會修改 EH/EMM DB 語意的工作，必須先建立獨立提案與 fixture / 指紋驗證。

*   **發布流程後續**:
    *   若要加入 zip、安裝包、正式發布檢查清單或自動產物整理，需另開 `RELEASE-PACK` 後續議題。
    *   GitHub 源碼包應維持純淨，不包含 `db.ast.json`、Everything SDK DLL、使用者設定、真實 DB、cache 或 log。

*   **長期目標**:
    *   持續降低 `gui/main_window.py` 的業務邏輯密度，逐步朝 Controller / View 分離推進。
    *   評估更完整的 GUI 自動化測試或大型樣本壓力測試，但不得直接塞進既有指紋巡檢範圍。

### **二、v17.1.0 架構與實現細節**

#### **1. 數據持久層與快取系統 (SQLite Cache Layer)**
*   **現況**: 圖片快取、資料夾狀態快取與掃描結果仍以 SQLite 為主，並配合 WAL / SQLite family 管理。
*   **可觀察性**: 掃描完成後會回報快取命中、重算、淘汰與重掃資料夾數量，協助判斷本輪是否發生大量重算。
*   **清理入口**: 設定頁提供圖片快取、廣告快取與資料夾掃描索引清理；清理前應有明確警告，並完整處理 `.db`、`-wal`、`-shm`。
*   **注意**: 不可直接刪除真實 cache 或 DB 進行測試；測試應使用 temp dir、fixture 或指紋巡檢。

#### **2. 核心比對與掃描流程 (Core Engine / Scanner)**
*   **核心流程**: `core_engine.py`、`core/similarity_flow.py`、`processors/scanner.py` 共同負責圖片掃描、hash 計算、快取復用與相似度流程。
*   **效能重點**: Phase A/B/C/E 已導入向量化與 lazy feature 補算；廣告比對包含 LSH 與 Grid-Block Fallback。
*   **安全邊界**: hash 流程需保留 0-hash 過濾、自身比對 guard、候選數量 cap 與缺 wHash 時的既有語意。
*   **Everything SDK**: 主掃描可透過 `processors/everything_ipc.py` 借用本機 Everything 服務；服務不可用時必須安全回退。

#### **3. 外掛系統 (Plugin System)**
*   **設定自動化**: `BasePlugin` 提供 `get_default_config()`，主程式會收集外掛設定並合併至 `config.json`。
*   **資源共享**: 外掛應優先呼叫核心流程或公開 API，不應私自維護重複掃描與 hash 計算邏輯。
*   **相似卷宗外掛**: `plugins/manga_deduplication/` 依賴 imagehash / pyzbar 等 runtime，打包時需確保對應二進位依賴存在。
*   **EH 數據庫外掛**: `plugins/eh_database_tools/` 負責 EH/EMM DB 同步、Pagediff 修復、備份輪轉與可選 EMM/CDP 自動化。任何修改都必須尊重 `file_deleted / non-tag / exist / hiddenBook` 的狀態語意。

#### **4. 日誌與診斷系統 (Logs / Diagnostics)**
*   **Runtime LOG**: 主流程任務結束時會輸出固定 `# RECAP` 摘要區塊，方便快速判讀結果。
*   **累積 LOG**: `info_log.txt` / `error_log.txt` 已加入大小輪轉，避免無限膨脹。
*   **EH 診斷**: EH 同步包含 `[EH timing]` 與細粒度進度；EXE 版 EMM subprocess 會輸出 `data/logs/emm_subprocess.log`。
*   **AI 讀 LOG 原則**: 先讀摘要、指紋與 RECAP，再決定是否展開完整 LOG，避免不必要的上下文消耗。

#### **5. 專案結構 (Directory Structure)**
v17.1.0 的主要結構如下：

```text
project_root/
├── app.py                         # 程式入口
├── config.py                      # 全域設定、版本與 runtime/resource 路徑
├── core_engine.py                 # 核心 hash 與比對入口
├── core/
│   ├── cache_flow.py              # 快取流程與統計
│   └── similarity_flow.py         # 相似度候選、過濾與比對流程
├── gui/
│   ├── main_window.py             # 主視窗、任務流程與結果呈現
│   ├── settings_window.py         # 設定視窗與外掛設定 UI
│   └── components/
│       └── tooltip.py
├── processors/
│   ├── scanner.py                 # 掃描、快取與 Everything 整合
│   ├── everything_ipc.py          # Everything SDK IPC 包裝
│   └── qr_engine.py               # QR 與 hash 輔助流程
├── plugins/
│   ├── manga_deduplication/
│   └── eh_database_tools/
├── data/                          # runtime 資料；源碼包不應包含私人 DB/cache/log
├── scratch/                       # 開發與安全網腳本；不屬於發布版
└── UpdateRecords/                 # 已完成提案與歷史紀錄；不屬於發布版
```

### **三、模組化協作原則 (Updated)**

#### **核心模組職責劃分**

*   **`gui/main_window.py`**
    *   **職責**: 負責主視窗佈局、任務流程接線、狀態列、進度條、結果呈現與 RECAP 輸出。
    *   **修改原則**: 儘量僅處理 UI 與流程協調。新增業務規則時，應優先放入 `core/`、`processors/` 或外掛模組。

*   **`gui/settings_window.py`**
    *   **職責**: 負責設定介面、外掛設定 UI、快取清理入口與 `config.json` 保存。
    *   **注意**: 設定文案需反映實際行為。例如圖片指紋目前以固定 64-bit base pHash 為基準，不得暗示未實作的解析度切換。

*   **`core_engine.py` / `core/`**
    *   **職責**: 提供 hash 計算、快取流程、相似度比對與候選過濾邏輯。
    *   **修改原則**: 保持公開 API 穩定。任何相似度公式、候選策略、hash 欄位語意或快取 schema 的修改，都需附 smoke / fingerprint / 實測 LOG 驗證。

*   **`processors/`**
    *   **職責**: 負責掃描、Everything IPC、QR 與底層資料處理。
    *   **修改原則**: Everything SDK 必須保留 fallback；QR / pyzbar / OpenCV 依賴不可讓整個主程式崩潰。

*   **`plugins/`**
    *   **命名規範**: 外掛設定鍵值 (`config` key) 必須加上前綴，避免與主程式或其他外掛衝突。
    *   **路徑規範**: 外掛不可自行猜測專案路徑，必須引用 `config.py` 的路徑常數或使用傳入 config。
    *   **DB 安全**: EH/EMM DB 相關改動必須先用 fixture 或 Level 3 指紋驗證；不得直接以真實 DB 測試破壞性操作。

*   **`scratch/`**
    *   **職責**: 存放開發診斷、安全網、打包與 smoke 測試腳本。
    *   **注意**: `scratch/` 不屬於發布版；若腳本會寫入資料，必須使用 temp dir 或明確隔離輸出。

*   **`UpdateRecords/`**
    *   **職責**: 存放已完成提案、歷史紀錄與歸檔資料。
    *   **注意**: 完成的 MD 應歸檔至 `UpdateRecords/completed-YYYY-MM/`，根目錄只保留當前必要文件。

*   **發布版資料規範**
    *   GitHub 源碼包不包含 `config.json`、真實 EH/EMM DB、cache、logs、fingerprints、`db.ast.json` 或 Everything SDK DLL。
    *   PyInstaller 預編譯 exe 可依打包環境納入 Everything DLL 與必要 runtime，但不得帶入使用者私人資料。

**交接結語：**
v17.1.0 是一個以效能、可觀察性、EH/EMM 安全性與發布穩定性為核心的穩定版本。接下來的開發應避免直接堆疊功能，優先維持「先提案、再受限實作、最後指紋/LOG 驗證」的流程。若要繼續優化 EH、掃描或打包流程，請先建立獨立 MD，說明問題、風險、回滾與驗證方式，再進入實作。
